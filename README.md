# منصّة إدارة المشاريع — علامة (Alaamaa-PMO)

داشبورد حوكمة المشاريع بين مكتب إدارة المشاريع (PMO) في علامة وعملائها.

- **النشر:** Cloudflare Pages على `pmo.alaamaa.com` (تلقائي عند الدفع لـ`main`)
- **القاعدة والمصادقة:** Supabase (مشترك مع الأدمن، جداول `pmo_` معزولة)
- **الدخول:** Google (حساب علامة)

## الأدوار والصلاحيات
- **PMO (admin):** بناء وتعديل الخطة، اعتماد Baseline، بتّ طلبات التغيير، إدارة وصول العملاء، تحويل العملاء المحتملين.
- **فريق التسويق (manager):** تحديث الحالة والتقدّم والمتطلبات، رفع طلبات تغيير.
- **العميل:** عرض مشروعه فقط (لوحة قيادة، جانت، مخرجات)، رفع طلبات، النقاش. (عزل بـRLS)

## البنية المعمارية (مهم للتطوير)

التطوير يتم في `src/`، و`build.py` يدمجها في `index.html` واحد للنشر. **لا تحرّر `index.html` يدويًا** — فهو مُولّد.

```
src/
├── index.html   ← الهيكل (HTML) فقط — يحمّل الوحدات بالترتيب
├── styles.css   ← نظام التصميم (متغيّرات، ظلال، حالات، استجابة)
├── config.js    ← الإعدادات والثوابت (Supabase، الألوان، الأدوار، الصلاحيات PERMS)
├── engine.js    ← محرك CPM الخالص (scheduleTasks + computeTracking) — بلا تبعيات
├── api.js       ← طبقة البيانات: كل وصول لـSupabase معزول هنا حصرًا
├── views.js     ← دوال العرض (render + v*) وربط أحداثها (bind*)
└── app.js       ← الحالة، النوافذ (toast/dialog)، الشاشات، المعالجات
```

### مبدأ معماري: عزل طبقة البيانات
**كل** استدعاء لـSupabase (`sb.from`, `sb.rpc`, `sb.auth`) موجود في `api.js` فقط. بقية الملفات تستدعي دوال `api` (مثل `fetchPortfolio`, `updateTaskFields`, `insertCR`). عند تغيير مخطّط جدول، التعديل في مكان واحد.

### ترتيب تحميل الوحدات
`config` → `engine` → `api` → `views` → `app` (التبعيات من الأعلى للأسفل).

## سير العمل
1. عدّل ملفات `src/`.
2. شغّل `python build.py` → يولّد `index.html`.
3. ادفع لـ`main` → Cloudflare ينشر تلقائيًا.

## التحقق قبل الدفع
- توازن الأقواس + `node --check` على JS المُولّد.
- محاكاة تحميل الوحدات (vm) للتأكد من الترابط.
- اختبار تغييرات القاعدة ضمن معاملة متراجَع عنها.

راجع `ROADMAP.md` للمراحل والميزات.
